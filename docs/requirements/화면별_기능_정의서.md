# WorkShield 화면별 기능 정의서

## 1. 문서 개요

### 1.1 목적

WorkShield의 요구사항 정의서, User Flow, 상태 정의서, 운영 정책, 화면 설계 초안을 기반으로 프론트엔드·백엔드·MCP·LLM·기술리드가 화면 기능과 데이터 계약을 함께 이해하기 위한 문서다.

### 1.2 MVP 전체 흐름

1차 시연은 PDF 파일 1건을 실제 MCP로 검토해 업로드부터 결과 표시까지 검증한다. 운영 환경에서는 여러 사용자가 동시에 각자의 파일 1건을 검토할 수 있다.

```text
S01 계약서 업로드
  → S02 계약 유형 확인
  → S03 범위 외 안내(조건부)
  → S04 검토 진행
  → S05 검토 결과
  → S06 조항 상세
  ├→ S07 검토 결과 챗봇
  └→ S08 협의 문구 제안
오류·빈 문서·세션 만료 → S09 오류·중단 안내
```

### 1.3 공통 원칙

- 지원 계약 유형은 SW 프리랜서 용역, SI 하도급, SM 하도급 3가지로 고정한다.
- 사용자가 SW 프리랜서 용역, SI 하도급, SM 하도급 중 하나를 직접 선택하거나 범위 외 진행을 명시적으로 확인해야 검토를 시작할 수 있다.
- 결과는 법적 판단이 아닌 검토 후보와 참고 설명으로 표현한다.
- 결과가 비어 있어도 “문제 없음”으로 표시하지 않는다.
- 계약서 원문, 대화 이력, 협의 문구는 영구 저장하지 않는다.
- 화면에는 MCP 원본 코드와 사용자용 표시명을 모두 활용하되, 사용자에게는 이해하기 쉬운 표현을 우선한다.

## 2. 우선순위 및 역할

| 우선순위 | 기준 |
|---|---|
| P0 | 업로드부터 검토 결과 확인까지의 핵심 흐름과 필수 예외 처리 |
| P0 | 단일 협의 문구 생성·복사 |
| P1 | 결과 기반 챗봇, 근거 표시, 주의 문구 |
| P2 | 복수 대안, 생성 방향 선택, 레드라인, 편집, 재검토, 고도화된 취소 |

역할 배정 전까지는 담당자를 다음처럼 표기한다.

- 프론트: 화면, 입력, 상태 표현, 사용자 인터랙션
- 백엔드: 세션, API, 상태 전이, DTO 정규화, 임시 데이터 관리
- MCP: 문서 파싱·유형 분석·검토·근거 조회 연동
- LLM: 챗봇·협의 문구 생성 및 구조화 출력
- 기술리드: 배포, 보안, 제한값, 로그·삭제 정책, 전체 기술 계약 조율

## 3. 공통 기능 규칙

### 3.1 상태 처리 우선순위

1. 최상위 처리 상태(`status`) 확인
2. 오류·세션 만료·재시도 가능 여부 확인
3. 정상인 경우에만 하위 결과 배열 표시
4. 결과 배열이 비어 있으면 별도 빈 상태 표시

### 3.2 공통 오류 규칙

- 확장자, 실제 형식, 크기, 손상, 암호화, 빈 문서는 MCP 호출 전에 차단한다.
- 오류 유형별로 원인, 사용자 영향, 재시도 가능 여부, 다음 행동을 표시한다.
- retryable이 아닌 오류에는 재시도 버튼을 제공하지 않는다.
- 동일 요청의 중복 실행을 방지한다.
- 내부 접속정보, API 키, 계약서 원문, 프롬프트 본문을 오류 화면이나 로그에 노출하지 않는다.

### 3.3 공통 표기 규칙

- 법률 판단 표현: 사용하지 않음
- 권장 표현: “확인이 필요합니다”, “차이가 있는 후보입니다”, “관련 표현으로 보입니다”
- `MISSING`: 경고가 아니라 “포함 여부 확인” 체크리스트로 표시
- `toxic_patterns`: “주의 문구 후보”로 표시
- 매칭 점수·신뢰도 숫자: MVP 화면에 표시하지 않음

### 4.2 기능 목록

| 기능 ID | 기능 | 설명 | 우선순위 |
|---|---|---|---|
| S01-F01 | 파일 선택 | 파일 선택 버튼 또는 드래그 앤 드롭으로 파일을 선택한다. | P0 |

| S01-F04 | 업로드 진행 표시 | 파일명, 용량, 업로드 진행률, 성공·실패 상태를 표시한다. | P0 |

| S01-F06 | 검토 시작 | 검증이 통과된 파일만 업로드 및 다음 단계로 이동한다. | P0 |

### 4.3 상태별 동작

| 상태 | 화면 동작 |
|---|---|
| `READY` | 파일 선택 가능, 검토 시작 버튼은 유효 파일 선택 전 비활성화 |
| `UPLOADING` | 진행률 표시, 파일 교체·중복 제출 제한 |
| 업로드 성공 | 계약 유형 분석 상태로 전환하고 S02로 이동 |
| 검증 실패 | 실패 원인을 유형별로 표시하고 재업로드 유도 |
| `FAILED` | 재시도 가능 여부에 따라 새 파일 업로드 또는 안내만 제공 |

### 4.4 입력 및 데이터

| 데이터 | 필수 | 설명 |
|---|---:|---|
| `file_name` | 예 | 원본 파일명 표시용 |
| `file_size` | 예 | 제한값 및 화면 표시용 |
| `file_type` | 예 | 확장자와 실제 형식 검증 |
| `upload_session_id` | 예 | 업로드 요청 추적용 |

### 4.5 검증 및 예외

- 확장자가 없거나 지원하지 않는 파일은 MCP를 호출하지 않는다.
- 확장자와 실제 형식이 다르면 별도 오류로 표시한다.
- 손상·암호화·빈 문서는 “검토 결과 없음”이 아니라 재업로드 안내를 표시한다.
- 백엔드 검증을 최종 기준으로 사용한다.

### 4.6 협업/API 확인사항

- 최대 파일 크기 및 요청 빈도 제한값
- 업로드 진행 이벤트 제공 방식
- `file_path` 단독 입력 또는 `file_content + file_name` 쌍 입력 계약
- 업로드 실패 시 `retryable` 표준

관련 요구사항: FR-1-1~FR-1-3, FR-1-11, FR-7-1, FR-7-9, FR-7-10.

### 4.8 기능 목록

|---|---|---|---|

| S02-F02 | 사용자 선택 | 사용자가 세 가지 유형 중 하나를 직접 선택한다. | P0 |
| S02-F03 | 검토 시작 | 선택 유형이 확정된 경우에만 검토를 시작한다. | P0 |

### 4.9 상태별 동작

| 상태 | 화면 동작 |
|---|---|
| `ANALYZING_CONTRACT_TYPE` | 분석 중 표시, 유형 선택 및 검토 시작 비활성화 |
| `TYPE_SELECTION_REQUIRED` | SW 프리랜서 용역, SI 하도급, SM 하도급을 표시하고 사용자 선택을 요구 |
| `CONTRACT_TYPE_UNCERTAIN` | 사용자가 유형을 선택하기 전까지 검토 진행 차단 |
| `READY_TO_REVIEW` | 최종 선택 유형과 선택 경로를 확인하고 검토 시작 허용 |
| 오류 | 오류 원인, 재시도 가능 여부, 새 업로드 선택지 표시 |

### 4.10 데이터 계약

```json
{
  "scope_status": "IN_SCOPE|CONTRACT_TYPE_UNCERTAIN|OUT_OF_SCOPE|EMPTY_DOCUMENT",
  "selected_contract_type": "string|null",
  "selection_source": "manual|null"
}
```

### 4.11 검증 규칙

- 지원 유형은 SW 프리랜서 용역(`SW_FREELANCE`), SI 하도급(`SI_SUBCONTRACT`), SM 하도급(`SM_SUBCONTRACT`) 3가지로 고정하며 사용자가 직접 선택한 값을 최종 선택값으로 보관한다.
- 유형 선택 전 `review_requested` 요청을 차단한다.
- `EMPTY_DOCUMENT`이면 `review_contract`를 호출하지 않는다.

관련 요구사항: FR-1-4~FR-1-6, FR-1-8~FR-1-10.

## S03. 범위 외 안내

### 4.12 기본 정보

| 항목 | 정의 |
|---|---|
| 화면 ID | S03 |
| 화면명 | 범위 외 안내 |
| 표현 | S02 화면 내 안내 영역 또는 모달 |
| 우선순위 | P0 |
| 주요 역할 | 프론트·백엔드·MCP |

### 4.13 기능 정의

- 범위 외 상태와 감지된 문서 유형을 표시한다.
- 계속 진행 시 결과의 제한사항을 안내한다.
- `다른 문서 선택`은 현재 임시 데이터를 정리하고 `READY`로 전환한다.
- `계속 진행`은 사용자의 명시적 확인 이력을 저장하고 `READY_TO_REVIEW`로 전환한다.
- 명시적 확인 없이 자동으로 검토를 시작하지 않는다.

권장 문구: “지원 범위 밖 계약으로 분석될 수 있습니다. 결과는 참고용 검토 후보이며 정확한 판단을 보장하지 않습니다. 계속 진행하시겠습니까?”

관련 요구사항: FR-1-7.

|---|---|---|

| S04-F02 | 진행률 표시 | 서버의 실제 진행 이벤트를 기반으로 진행률을 표시한다. |

| S04-F05 | 중복 실행 방지 | 동일 검토 요청의 중복 제출을 제한한다. |

### 4.16 상태 및 이벤트

| 단계 | 사용자 문구 |
|---|---|
| `PREPARE` | 검토를 준비하고 있습니다 |
| `BATCH_SEARCH` | 비교 기준 조항을 찾고 있습니다 |
| `RERANK` | 비교할 조항을 선별하고 있습니다 |
| `CLAUSE_REVIEW` | 계약 조항을 비교하고 있습니다 |
| `MISSING_DETECTION` | 포함 여부를 확인할 표준조항을 찾고 있습니다 |
| `RESULT_ASSEMBLY` | 결과를 정리하고 있습니다 |

### 4.17 구현 규칙

- `QUEUED`는 향후 작업 대기열 확장을 위한 상태로 유지한다. 이번 프로젝트에서는 별도 작업 큐를 구현하지 않고 검토 요청 접수 직후 `REVIEWING`으로 전환한다.
- 여러 사용자의 검토 요청을 독립적인 `review_id`로 처리하며, 동일 요청의 중복 실행은 방지한다.
- 프론트 임의 타이머를 사용하지 않는다.
- 상태 조회 API를 2~3초 간격으로 호출하는 방식을 기본으로 사용한다. `QUEUED`, `REVIEWING`에서만 조회하고 `COMPLETED`, `FAILED`, `EXPIRED`이면 종료한다. SSE는 보조 수단으로 사용한다.
- 진행률은 역행하지 않는다.
- SSE 연결이 끊기면 상태 조회 API로 최신 상태를 확인한다.
- MVP 취소는 실제 MCP 작업 취소를 보장하지 않으며, 취소 후 결과를 폐기하고 임시 데이터를 정리한다.

관련 요구사항: FR-8-1~FR-8-6.

### 4.20 결과 상태 표시

| 코드 | 사용자 표시명 | 설명 |
|---|---|---|
| `NONE` | 대응 표준조항 있음 | 대응하는 표준조항 후보가 있음 |
| `EXTRA` | 추가·변형 내용 확인 | 표준 대비 추가·변경된 내용 후보 |
| `NO_MATCH` | 대응 조항 확인 필요 | 사용자 조항에서 대응 표준조항 후보를 찾기 어려움 |
| `MISSING` | 포함 여부 확인 | 표준조항이 계약서 전체에서 확인되지 않아 체크가 필요함 |

### 4.21 표현 및 예외 규칙

- `MISSING`을 경고 또는 오류 색상으로 표현하지 않는다.
- `NO_MATCH`는 사용자 조항에서 표준조항을 찾는 방향이다.
- `MISSING`은 표준조항에서 계약서 전체를 확인하는 방향이다.
- `MISSING`은 사용자 조항이 비어 있을 수 있으므로 빈 값을 오류로 처리하지 않는다.
- `toxic_patterns`가 없어도 안전하다는 의미로 표시하지 않는다.
- 결과 배열이 비어 있으면 “데이터가 없습니다”와 다음 행동을 표시한다.
- 최상위 `CORPUS_UNAVAILABLE`, `INVALID_CONFIG`, `PIPELINE_ERROR`를 먼저 처리한다.
- `MISSING` 건수는 일반 조항 결과 목록 수에 포함하지 않으며, 별도 체크리스트 목록 수와 일치시킨다.

관련 요구사항: FR-2-1~FR-2-13, FR-3-1~FR-3-6, FR-4-1~FR-4-5.

- 재시도 가능 여부에 따라 버튼을 조건부 표시한다.
- 재시도 시 동일 요청의 중복 실행을 방지한다.
- 재시도는 MCP 연결 실패, 타임아웃, `CORPUS_UNAVAILABLE`, `PIPELINE_ERROR`에만 제공한다. 파일 검증 실패, 빈 문서, 계약 유형 미선택, 범위 외 미확인, `INVALID_CONFIG`, `EXPIRED`에는 제공하지 않는다.
- 재시도할 수 없는 실패는 즉시 삭제한다. 검토 완료 후에는 결과 페이지 이용 중 유지하며, 페이지 이탈·새 검토 시작·명시적 종료 시 삭제를 요청한다. 요청 누락은 마지막 사용자 액션 후 30분 TTL로 보완하고, 검토 진행 중에는 검토 완료 전까지 세션을 만료시키지 않는다. 세션 만료 시 `EXPIRED` 처리 후 파일과 결과를 삭제한다.

## 7. QA 시나리오

### 7.1 정상 흐름

- [ ] 지원 형식 파일을 업로드한다.
- [ ] SW 프리랜서 용역, SI 하도급, SM 하도급 중 하나를 직접 선택한다.
- [ ] 실제 서버 진행 이벤트가 화면에 표시된다.
- [ ] 결과 요약과 목록 수가 일치한다.
- [ ] 조항 상세에서 사용자 조항·표준조항·근거를 확인한다.
- [ ] 챗봇 답변에 검증된 출처가 표시된다.
- [ ] 충분한 근거가 있는 조항에서 협의 문구를 생성하고 복사한다.

### 7.2 예외 흐름

- [ ] 지원하지 않는 파일이 MCP 호출 전에 차단된다.
- [ ] 손상·암호화·빈 문서가 구분 안내된다.
- [ ] `CONTRACT_TYPE_UNCERTAIN`에서 유형 선택 전 검토가 차단된다.
- [ ] `OUT_OF_SCOPE`에서 명시적 확인 전 검토가 차단된다.
- [ ] `MISSING`이 경고가 아닌 체크리스트로 표시된다.
- [ ] 빈 결과가 “문제 없음”으로 표시되지 않는다.
- [ ] 챗봇 근거 부족 시 추측 답변 대신 거절된다.
- [ ] 협의 문구 근거 부족 시 일반 문구가 생성되지 않는다.
- [ ] 세션 만료 후 새 검토로 이동하고 임시 데이터가 정리된다.
