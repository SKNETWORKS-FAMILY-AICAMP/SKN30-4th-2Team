# WorkShield MVP 사용자 흐름·상태 정의서 초안

## 1. MVP 전체 흐름

```text
계약서 업로드
→ 계약 유형 후보 분석
→ 사용자가 비교 기준 확정
→ 계약서 검토 진행
→ 결과 확인
→ 조항 상세·법령 근거 확인
→ 챗봇 질문
→ 수정·협의 문구 생성·복사
```

## 2. 핵심 원칙

- 지원 계약 유형은 SW 프리랜서 용역, SI 하도급, SM 하도급 3가지로 고정한다. 사용자는 세 가지 유형 중 하나를 직접 선택한다.
- 사용자가 최종 선택한 계약 유형을 기준으로 검토한다.
- 검토 결과와 챗봇 답변은 법적 판단이 아닌 검토 후보·참고 설명으로 제공한다.
- 재시도할 수 없는 실패는 즉시 삭제한다. 검토 완료 후에는 결과 페이지 이용 중 유지하고, 페이지 이탈·새 검토 시작·명시적 종료 시 삭제 요청한다. 요청 누락은 마지막 사용자 액션 후 30분 TTL로 보완하며, 검토 진행 중에는 검토 완료 전까지 세션을 만료시키지 않는다. 세션 만료 시 EXPIRED 처리 후 파일과 결과를 삭제한다.
- 로그인 없이 서버가 발급한 추측 불가능한 세션 토큰으로 세션·검토 결과 접근을 제한한다. 세션 및 임시 파일 TTL은 마지막 사용자 액션 후 30분이다.
- 챗봇과 수정 문구는 현재 검토 결과와 관련 근거가 있을 때만 제공한다.

## 3. 주요 화면

| 화면 ID | 화면명 | MVP 기능 |
|---|---|---|
| S01 | 계약서 업로드 | 파일 선택, 파일 검증, 업로드 진행 |
| S02 | 계약 유형 확인 | 추천 유형·후보 표시, 사용자 최종 선택 |
| S03 | 범위 외 안내 | 범위 외 계약 경고, 계속 진행 재확인 |
| S04 | 검토 진행 | 실제 진행 단계·진행률, 오류·재시도 |
| S05 | 검토 결과 | 결과 요약, 목록, 검색·필터, 주의 문구 후보 |
| S06 | 조항 상세 | 사용자 조항, 표준조항, 결과 설명, 법령 근거 |
| S07 | 검토 결과 챗봇 | 결과 기반 질문, 답변, 출처 |
| S08 | 수정·협의 문구 | 제안 생성, 근거 확인, 복사 |
| S09 | 오류·중단 안내 | 오류 원인, 재시도, 재업로드 |

## 4. 검토 상태 목록

| 상태 코드 | 의미 | 연결 화면 |
|---|---|---|
| `READY` | 파일을 선택할 수 있는 초기 상태 | S01 |
| `UPLOADING` | 파일 업로드 중 | S01 |
| `ANALYZING_CONTRACT_TYPE` | 계약 유형 후보 분석 중 | S02 |
| `TYPE_SELECTION_REQUIRED` | 사용자의 계약 유형 선택이 필요함 | S02 |
| `OUT_OF_SCOPE_CONFIRMATION_REQUIRED` | 범위 외 계약에 대한 재확인이 필요함 | S03 |
| `READY_TO_REVIEW` | 사용자 선택이 완료되어 검토 가능 | S02 |
| `QUEUED` | 검토 요청이 접수된 직후 `REVIEWING`으로 전환되는 확장용 상태 | S04 |
| `REVIEWING` | 계약서 검토 중 | S04 |
| `COMPLETED` | 검토 결과 생성 완료 | S05 |
| `FAILED` | 업로드·분석·검토 처리 실패 | S09 |
| `EXPIRED` | 검토 세션 만료 | S09 |

## 5. 계약서 검토 상태 전이표

| 현재 상태 | 사용자 행동 | 시스템 처리 | 다음 상태 |
|---|---|---|---|
| `READY` | 파일 선택·업로드 요청 | 확장자, 크기, MIME 등 기본 검증 | `UPLOADING` |
| `UPLOADING` | 대기 | 파일 수신 및 문서 처리 | `ANALYZING_CONTRACT_TYPE` |
| `ANALYZING_CONTRACT_TYPE` | 대기 | `assess_contract_scope`로 후보 분석 | `TYPE_SELECTION_REQUIRED` |
| `TYPE_SELECTION_REQUIRED` | 계약 유형 선택 | 선택 유형과 선택 경로 저장 | `READY_TO_REVIEW` |
| `OUT_OF_SCOPE_CONFIRMATION_REQUIRED` | 계속 진행 확인 | 확인 이력 저장 | `READY_TO_REVIEW` |
| `READY_TO_REVIEW` | 검토 시작 | 선택 유형으로 검토 요청을 접수 | `QUEUED` |
| `QUEUED` | 즉시 전환 | 별도 작업 큐 없이 검토 작업 시작 | `REVIEWING` |
| `QUEUED` | 대기 | 검토 작업 접수·실행 실패 | `FAILED` |
| `REVIEWING` | 대기 | 조항 비교·누락 후보·주의 문구 분석 | `COMPLETED` |
| `COMPLETED` | 결과·상세 조회 | 결과와 관련 근거 제공 | `COMPLETED` |

## 6. 계약 유형 분석 분기

| MCP 분석 결과 | 다음 상태 | 사용자 화면·행동 |
|---|---|---|
| `IN_SCOPE` | `TYPE_SELECTION_REQUIRED` | 추천 유형과 후보 확인 후 최종 선택 |
| `CONTRACT_TYPE_UNCERTAIN` | `TYPE_SELECTION_REQUIRED` | 후보 또는 전체 지원 유형 중 직접 선택 |
| `OUT_OF_SCOPE` | `OUT_OF_SCOPE_CONFIRMATION_REQUIRED` | 범위 제한 안내 후 계속 진행 또는 취소 |
| `EMPTY_DOCUMENT` | `FAILED` | 빈 문서·스캔본·손상 가능성 안내 후 재업로드 |
| 오류 | `FAILED` | 오류 안내 및 재시도 |

권장 화면 문구:

> 문서 내용을 바탕으로 비교할 표준계약서 후보를 제안합니다. 실제 계약 관계를 가장 잘 설명하는 기준을 선택해 주세요.

## 7. 검토 진행 상태

| 진행 단계 | 화면 문구 예시 |
|---|---|
| `PREPARE` | 검토를 준비하고 있습니다 |
| `BATCH_SEARCH` | 관련 표준조항을 찾고 있습니다 |
| `RERANK` | 비교할 조항을 선별하고 있습니다 |
| `CLAUSE_REVIEW` | 계약 조항을 비교하고 있습니다 |
| `MISSING_DETECTION` | 확인할 표준조항을 찾고 있습니다 |
| `RESULT_ASSEMBLY` | 결과를 정리하고 있습니다 |

- 진행률은 프론트엔드 임의 타이머가 아니라 서버의 실제 진행 이벤트를 사용한다.
- 진행 이벤트는 `review_id`와 연결한다.
- 상태 조회 API를 2~3초 간격으로 호출하는 방식을 기본으로 사용한다. `QUEUED`, `REVIEWING`에서만 조회하며 `COMPLETED`, `FAILED`, `EXPIRED`면 종료한다. SSE는 보조 수단으로 사용한다.
- `QUEUED`는 향후 작업 대기열 확장을 위한 상태로 유지한다. 이번 프로젝트에서는 별도 작업 큐를 구현하지 않으며, 검토 요청 후 즉시 `REVIEWING`으로 전환한다.
- 검토 중 오류가 발생하면 진행 표시를 종료하고 오류·재시도 가능 여부를 표시한다.

## 8. 검토 결과 상태

| 결과 코드 | 사용자 표시명 예시 | 의미 |
|---|---|---|
| `NONE` | 대응 표준조항 있음 | 대응하는 표준조항 후보가 있음 |
| `EXTRA` | 추가·변형 내용 확인 | 표준 대비 추가·변경된 내용 후보 |
| `NO_MATCH` | 대응 조항 확인 필요 | 사용자 조항에서 대응 표준조항 후보를 찾기 어려움 |
| `MISSING` | 포함 여부 확인 필요 | 표준조항 관점에서 계약서 전체에 대응 내용이 보이지 않음 |

- `MISSING`은 경고 결과가 아니라 별도 체크리스트 영역으로 표시한다.
- `toxic_patterns`는 결과 코드와 별도로 “주의 문구 후보” 영역에 표시한다.
- 빈 결과를 “문제 없음”으로 표현하지 않는다.
- 매칭 점수와 신뢰도 수치는 MVP에서 표시하지 않는다.
- 요약 카드는 `NONE`, `EXTRA`, `NO_MATCH`와 `MISSING`을 집계상 분리한다. `MISSING`은 표준 체크리스트의 별도 건수로 표시한다.

## 9. 챗봇 상태 목록

챗봇 상태는 계약서 검토 상태와 별도로 관리한다.

| 상태 코드 | 의미 |
|---|---|
| `DISABLED` | 검토 결과가 없어 챗봇 사용 불가 |
| `READY` | 질문 입력 가능 |
| `ANSWERING` | 답변 생성 및 근거 검증 중 |
| `ANSWERED` | 답변 생성 완료 |
| `REFUSED` | 범위 밖 질문 또는 근거 부족으로 답변 제한 |
| `ERROR` | MCP·LLM 호출 또는 출력 검증 실패 |
| `EXPIRED` | 검토 세션 만료로 대화 종료 |

## 10. 챗봇 상태 전이표

| 현재 상태 | 사용자 행동 | 시스템 처리 | 다음 상태 |
|---|---|---|---|
| `DISABLED` | 없음 | 검토가 완료되면 활성화 | `READY` |
| `READY` | 질문 입력 | 현재 계약서·검토 결과·관련 법령 범위 확인 | `ANSWERING` |
| `ANSWERING` | 대기 | 근거 조회, 구조화 답변 생성, 출처 검증 | `ANSWERED` |
| `ANSWERING` | 대기 | 답변 범위 밖이거나 근거 부족 | `REFUSED` |
| `ANSWERING` | 대기 | MCP·LLM 호출 또는 검증 실패 | `ERROR` |
| `ANSWERED` | 후속 질문 | 현재 세션의 대화 맥락으로 답변 생성 | `ANSWERING` |
| `REFUSED` | 질문 수정 | 수정 질문 처리 | `ANSWERING` |
| `ERROR` | 재시도 | 동일 질문 재처리 | `ANSWERING` |

## 11. 챗봇 MVP 범위

챗봇은 다음 질문에 답변한다.

- 현재 검토 결과의 의미
- 사용자 조항과 표준조항의 차이
- `NONE`, `EXTRA`, `NO_MATCH`, `MISSING` 상태 설명
- 관련 법령의 원문·조문·출처
- 특정 조항에서 확인할 사항

챗봇은 다음을 제한한다.

- 위법·합법, 유리·불리, 승소 가능성의 단정
- 현재 검토와 무관한 질문
- 현재 세션에 없는 계약 조항 또는 법령 출처의 생성
- 계약서·법령 원문에 포함된 명령문의 실행

실질적인 계약·법령 답변에는 조항 ID, 표준조항 ID, 법령명·조번호 중 하나 이상의 출처를 표시한다.

## 12. 수정·협의 문구 상태 목록

| 상태 코드 | 의미 |
|---|---|
| `IDLE` | 조항을 선택하지 않은 초기 상태 |
| `SOURCE_SELECTED` | 사용자 조항과 대응 표준조항을 선택함 |
| `GENERATING` | 수정·협의 문구 생성 중 |
| `GENERATED` | 제안 문구 생성 완료 |
| `INSUFFICIENT_GROUNDING` | 근거 부족 또는 필수값 누락 |
| `ERROR` | 생성 또는 검증 실패 |

## 13. 수정·협의 문구 상태 전이표

| 현재 상태 | 사용자 행동 | 시스템 처리 | 다음 상태 |
|---|---|---|---|
| `IDLE` | 결과 조항 선택 | 사용자 조항·표준조항·상태·카테고리 확인 | `SOURCE_SELECTED` |
| `SOURCE_SELECTED` | 제안 생성 요청 | 근거 기반 문구 생성 | `GENERATING` |
| `GENERATING` | 대기 | 생성 결과와 출처 검증 | `GENERATED` |
| `GENERATING` | 대기 | 근거 부족·필수값 누락 확인 | `INSUFFICIENT_GROUNDING` |
| `GENERATING` | 대기 | 생성·검증 실패 | `ERROR` |
| `GENERATED` | 복사 | 제안 문구 또는 문구+근거 복사 | `GENERATED` |

MVP에서는 다음만 제공한다.

- 단일 제안 문구 생성
- 생성 목적, 원문과의 주요 차이, 참고 표준조항 표시
- 복사 기능
- 계약서에 없는 금액·기간 등은 `[확인 필요]` 형태로 표시
- 원본 계약서 자동 수정 금지

## 14. MVP 이후 확장 기능

- 매칭 점수와 신뢰도 수치 표시
- LLM의 자유로운 추가 확인 의견
- 수정·협의 문구 생성 방향 선택
- 같은 조항의 복수 대안 문구
- 원문과 제안문의 레드라인 비교
- 제안 문구 직접 편집·임시 저장
- 수정 문구 재검토
- 검토 진행 취소

이 초안은 요구사항 정의서의 계약 유형 분석, 결과 표시, 챗봇, 수정·협의 문구, 진행 상태 요구사항을 MVP 기준으로 반영했습니다.
