# WorkShield 개발 협업용 화면설계서 초안

문서 목적: 제출용 시각 디자인 전 단계에서 프론트엔드·백엔드·MCP·LLM 구현 기준을 맞추기 위한 화면/상태/데이터 초안

기준 문서: `서비스_전체_흐름도.json`, `요구사항.json`, `운영_정책.json`, `사용자 흐름·상태 정의서 초안.md`

## 1. MVP 범위와 핵심 원칙

- 계약서 업로드 → 계약 유형 확인 → 계약서 검토 → 결과 확인 → 조항 상세/법령 근거 → 결과 기반 챗봇 → 수정·협의 문구 제안
- 1차 시연은 PDF 파일 1건을 실제 MCP로 검토해 업로드부터 결과 표시까지 확인한다. 운영 환경에서는 여러 사용자가 동시에 각자의 파일 1건을 검토할 수 있다.
- 지원 계약 유형은 SW 프리랜서 용역, SI 하도급, SM 하도급 3가지로 고정한다. 사용자는 세 가지 유형 중 하나를 직접 선택한다.
- 결과, 챗봇 답변, 협의 문구는 법률 자문이나 위법·합법 판단이 아닌 검토 후보/참고 설명이다.
- 계약서 원문·대화·제안 문구는 영구 저장하지 않는다. 완료·실패·취소·TTL 만료 후 임시 데이터를 정리한다.
- MVP에서 매칭 점수와 신뢰도 숫자는 노출하지 않는다.

## 2. 역할·우선순위 기준

역할 배정은 아직 전달받지 못했으므로 아래는 기능별 협업 대상이다. 실제 담당자 이름은 추후 `담당자` 열에 입력한다.

| 우선순위 | 범위 | 포함 기능 |
|---|---|---|
| P0 | 반드시 동작해야 하는 핵심 흐름 | 업로드/검증, 계약 유형 후보·선택, 검토 진행, 결과 목록·필터, 조항 상세, MISSING 체크리스트, 오류/재시도 |
| P1 | MVP 핵심 품질 및 차별 기능 | 결과 기반 챗봇, 근거/출처 표시, 주의 문구 후보 |
| P2 | MVP 이후 확장 | 방향 선택, 복수 대안, 레드라인, 편집/임시 저장, 수정 문구 재검토, 검토 취소 완성 |

## 3. 전체 화면 흐름

```text
S01 업로드
  └─ 업로드 성공 → S02 계약 유형 확인
       ├─ IN_SCOPE / UNCERTAIN → 사용자 유형 선택 → S04 검토 진행
       ├─ OUT_OF_SCOPE → S03 안내 → 명시적 계속 진행 → S04
       └─ EMPTY_DOCUMENT / 오류 → S09 오류·재업로드
S04 검토 진행 → 완료 → S05 결과
S05 결과 → S06 조항 상세 → S08 협의 문구 제안
          └→ S07 결과 기반 챗봇
```

## 4. 화면 목록

| ID | 화면명 | 경로/표현 | 우선순위 | 상태 |
|---|---|---|---|---|
| S01 | 계약서 업로드 | `/review` | P0 | READY, UPLOADING, FAILED |
| S02 | 계약 유형 확인 | `/review/type` | P0 | ANALYZING_CONTRACT_TYPE, TYPE_SELECTION_REQUIRED, READY_TO_REVIEW |
| S03 | 범위 외 안내 | `/review/type` 상태/모달 | P0 | OUT_OF_SCOPE_CONFIRMATION_REQUIRED |
| S04 | 검토 진행 | `/review/{review_id}/progress` | P0 | QUEUED, REVIEWING |
| S05 | 검토 결과 | `/review/{review_id}/results` | P0 | COMPLETED |
| S06 | 조항 상세 | 결과 화면 드로어 | P0 | 상세 조회 |
| S07 | 검토 결과 챗봇 | 결과 화면 하단 | P1 | DISABLED~EXPIRED |
| S08 | 협의 문구 제안 | 조항 상세 모달 | P0 | IDLE~ERROR |
| S09 | 오류·중단 안내 | 공통 상태 화면/영역 | P0 | FAILED, EXPIRED |

## 5. 화면별 상세 설계

### S01. 계약서 업로드

목적: 검토할 파일을 선택하고 서버 검증을 거쳐 업로드한다.

화면 구성:

- 파일 선택 영역: 지원 형식 HWP/HWPX/PDF/DOCX
- 선택 파일 정보: 파일명, 용량, 삭제/교체
- 업로드 버튼: 파일 검증 통과 전 비활성화
- 진행 상태: 업로드 진행률과 검토 진행률을 별도 표시
- 안내 문구: 계약서 원문과 결과는 영구 저장되지 않음

상태별 동작:

| 상태 | 표시/행동 |
|---|---|
| READY | 파일 선택 가능 |
| UPLOADING | 파일명·용량·업로드 진행률, 중복 업로드 방지 |
| 실패 | 확장자/실제 형식 불일치/손상/암호화/빈 문서/용량 초과를 구분 안내 |

검증 규칙: 프론트에서 1차 검증하되 백엔드 검증을 최종 기준으로 한다. 검증 실패 시 MCP를 호출하지 않는다.

협업/API 확인: 최대 파일 크기 10MB, 동시 검토 수 2건, 업로드 진행 이벤트 제공 여부, `file_path` 단독 또는 `file_content+file_name` 쌍 입력 계약, 서버 발급 세션 토큰.

관련 요구사항: FR-1-1~FR-1-3, FR-1-11, FR-7-1~FR-7-10.

### S02. 계약 유형 확인

목적: 시스템 추천 후보를 보여주고 사용자가 검토 기준을 최종 선택한다.

화면 구성:

- 추천 계약 유형 카드
- 후보 목록 및 SW 프리랜서 용역·SI 하도급·SM 하도급 선택지
- 추천 유형과 사용자 최종 선택을 구분
- 선택 근거/설명(응답으로 제공되는 경우만)
- `검토 시작` 버튼

상태별 동작:

- `ANALYZING_CONTRACT_TYPE`: 분석 중 표시, 선택 불가
- `TYPE_SELECTION_REQUIRED`: 선택 전 검토 시작 불가
- `CONTRACT_TYPE_UNCERTAIN`: 후보 또는 전체 유형에서 직접 선택
- `READY_TO_REVIEW`: 선택 유형과 선택 경로를 확인하고 검토 시작

백엔드 데이터: `scope_status`, `selected_contract_type`, `selection_source`. 지원 계약 유형은 SW 프리랜서 용역(`SW_FREELANCE`), SI 하도급(`SI_SUBCONTRACT`), SM 하도급(`SM_SUBCONTRACT`) 3가지이며, 사용자는 세 가지 유형 중 하나를 직접 선택한다.

관련 요구사항: FR-1-4~FR-1-6, FR-1-9~FR-1-10.

### S03. 범위 외 안내

목적: 지원 범위 밖 문서임을 알리고 사용자가 계속 진행할지 명시적으로 확인한다.

구성/행동:

- 범위 제한 안내 및 현재 감지 유형 표시
- `다른 문서 선택` → READY
- `계속 진행` → 확인 이력 저장 후 READY_TO_REVIEW
- 자동 진행 금지

문구 예시: “지원 범위 밖 계약으로 분석될 수 있습니다. 결과는 참고용 검토 후보이며 정확한 판단을 보장하지 않습니다. 계속 진행하시겠습니까?”

관련 요구사항: FR-1-7.

### S04. 검토 진행

목적: 서버 실제 이벤트 기반으로 검토 진행 상황을 보여준다.

화면 구성:

- 단계 목록: 준비 → 비교 기준 검색 → 조항 선별 → 조항 비교 → 포함 여부 확인 → 결과 정리
- 현재 단계, 진행률, 처리 조항 수/전체 수(제공될 때)
- `review_id` 연결 상태
- 오류 시 오류 메시지와 재시도 가능 여부

구현 협의: 프론트 임의 타이머 금지. 상태 조회 API를 2~3초 간격으로 호출하는 방식을 기본으로 하고 SSE를 보조 수단으로 사용하며 진행률 역행을 방지한다. `QUEUED`, `REVIEWING`에서만 조회하고 `COMPLETED`, `FAILED`, `EXPIRED`에서 종료한다. `QUEUED`는 향후 작업 대기열 확장을 위한 상태로 유지한다. 이번 프로젝트에서는 별도 작업 큐를 구현하지 않고 검토 요청 접수 직후 `REVIEWING`으로 전환한다. MVP 취소는 실제 MCP 작업 취소가 아니라 결과 폐기·임시 데이터 정리 정책임을 명시한다.

관련 요구사항: FR-8-1~FR-8-6.

### S05. 검토 결과

목적: 결과를 요약하고 사용자가 상태/카테고리/키워드로 탐색한다.

화면 구성:

- 상단: 선택한 계약 유형, 검토 안내, 법률 자문 아님 문구
- 요약 카드: `NONE`, `EXTRA`, `NO_MATCH` 조항 결과 건수
- 별도 체크리스트 요약: `MISSING` 건수
- 결과 목록: 조항 카드/테이블, 상태 표시명 + 원본 코드
- 필터: 상태, 카테고리, 키워드
- 별도 영역: “주의 문구 후보”
- 별도 영역: “표준 체크리스트(포함 여부 확인)”
- 결과 행 클릭 → S06
- 하단/측면 → S07 챗봇

표현 규칙:

- `MISSING`은 경고가 아니라 체크리스트 톤으로 표시하고, 표준조항에서 계약서 전체 방향으로 찾은 항목임을 설명한다.
- 결과 배열이 비어도 “문제 없음” 또는 “검토할 사항 없음”으로 표시하지 않는다.
- `toxic_patterns`가 없어도 “안전함”으로 해석되는 문구를 사용하지 않는다.
- 요약 수치와 실제 목록 수를 일치시킨다.
- `MISSING` 건수는 일반 조항 결과 목록에 합산하지 않고 표준 체크리스트 목록 수와 일치시킨다.

### 새로고침 복구 및 재시도 기준

- 프론트는 `session_id`, `review_id`, 현재 검토 경로를 임시 보관한다.
- 새로고침 후 저장된 값을 그대로 사용하지 않고 상태 조회 API로 서버 상태를 다시 확인한다.
- `QUEUED`·`REVIEWING`이면 진행 화면, `COMPLETED`이면 결과 화면, `FAILED`이면 오류·재시도 화면으로 복구한다.
- 재시도는 새 `review_id`를 발급하고 새 검토 진행 화면으로 이동한다.
- 재시도할 수 없는 실패는 즉시 삭제한다. 검토 완료 후에는 사용자가 결과 페이지를 이용하는 동안 유지한다. 페이지 이탈·새 검토 시작·명시적 종료 시 삭제를 요청하고, 브라우저 종료 등 요청이 누락되면 마지막 사용자 액션 후 30분 TTL로 삭제한다. 검토 진행 중에는 검토 완료 전까지 세션을 만료시키지 않는다. 세션 만료 시 `EXPIRED` 처리 후 파일과 결과를 삭제한다.

상태 오류: 최상위 `status`를 먼저 확인하고 `CORPUS_UNAVAILABLE`, `INVALID_CONFIG`, `PIPELINE_ERROR`별 안내와 재시도 가능 여부를 표시한다.

관련 요구사항: FR-2-1~FR-2-13, FR-3-1~FR-3-6, FR-4-1~FR-4-5.

### S06. 조항 상세

목적: 사용자 조항과 표준조항의 비교 결과, 근거, 다음 행동을 확인한다.

화면 구성:

- 사용자 조항 원문/조항 ID
- 대응 표준조항 원문/표준조항 ID
- 비교 상태 및 쉬운 설명
- 결과 방향성 설명: 사용자 조항→표준조항 또는 표준조항→계약서 전체
- 관련 법령: 법령명, 조번호, 본문, 출처를 구분
- `협의 문구 제안` 버튼: 근거가 모두 있는 경우 활성화

MISSING은 사용자 조항이 비어 있을 수 있으므로 빈 영역을 오류로 처리하지 않는다. “해당 표준조항이 계약서 전체에서 확인되지 않아 포함 여부 확인이 필요합니다”처럼 표시한다.

### S07. 검토 결과 챗봇

목적: 현재 계약서·검토 결과·관련 법령 근거에 한정해 질문에 답한다.

상태:

| 상태 | UI |
|---|---|
| DISABLED | 검토 완료 전 입력 비활성화 |
| READY | 질문 입력 가능 |
| ANSWERING | 답변 생성/근거 검증 중, 중복 전송 방지 |
| ANSWERED | 설명 + 출처 + 한계 표시 |
| REFUSED | 범위 밖/근거 부족 안내, 질문 수정 유도 |
| ERROR | 추측 답변 없이 오류와 재시도 표시 |
| EXPIRED | 세션 만료 및 새 검토 안내 |

답변 카드 구성: LLM 설명, 근거 출처(사용자 조항 ID/표준조항 ID/법령명·조번호 중 하나 이상), 법령 원문 카드, 제한사항. 법령 원문과 설명은 시각적으로 구분한다.

금지: 위법·합법, 유리·불리, 승소 가능성 단정; 현재 검토와 무관한 질문 답변; 세션에 없는 출처 생성. 대화 이력은 영구 저장하지 않는다.

백엔드 협의: `answer`, `refused`, `sources`, `limitations`, `tool_status` 구조화 응답, 세션 근거 ID 검증, MCP 실패 표준화.

관련 요구사항: FR-5-1~FR-5-11.

### S08. 협의 문구 제안

목적: 선택한 사용자 조항과 대응 표준조항을 근거로 참고용 제안 문구를 생성·복사한다.

상태:

- `IDLE`: 조항 미선택
- `SOURCE_SELECTED`: 사용자/표준 조항 선택 완료
- `GENERATING`: 생성 및 검증 중
- `GENERATED`: 제안 문구, 생성 목적, 주요 차이, 참고 표준조항 표시
- `INSUFFICIENT_GROUNDING`: 근거 부족/필수값 누락으로 생성하지 않음
- `ERROR`: 생성·검증 실패 및 재시도 안내

상단 고정 안내: “자동 반영되지 않는 참고 초안이며 법률 자문이 아닙니다.”

MVP: 단일 제안, 본문 복사, `[확인 필요]` 표시. 원본 계약서 자동 수정 금지. 계약서에 없는 금액·기간·비율·조건을 임의로 생성하지 않는다.

P2: 생성 방향 선택, 복수 대안, 레드라인, 편집/임시 저장, 재검토.

관련 요구사항: FR-6-1~FR-6-13 중 FR-6-1~FR-6-8을 MVP 우선 반영.

### S09. 오류·중단 안내

공통 오류 유형:

| 유형 | 사용자 안내 | 다음 행동 |
|---|---|---|
| 지원하지 않는 형식/크기 초과 | 지원 형식·제한 안내 | 다른 파일 업로드 |
| 손상/암호화/빈 문서 | 읽을 수 없는 원인 안내 | 읽을 수 있는 파일 재업로드 |
| `CORPUS_UNAVAILABLE` | 비교 기준 자료를 사용할 수 없음 | 재시도 가능 시 재시도 |
| `INVALID_CONFIG` | 검토 설정을 확인할 수 없음 | 관리자/재시도 안내 |
| `PIPELINE_ERROR` | 검토 처리 중 오류 | 재시도 가능 여부 표시 |
| 세션 만료 | 임시 검토 세션이 종료됨 | 새 검토 시작 |

내부 오류 상세·접속정보·계약서 본문은 노출하지 않는다. 재시도는 retryable일 때만 제공하고 동일 요청 중복 실행을 막는다.

## 8. 프론트·백엔드 전달용 최소 데이터 계약

### 검토 세션

```json
{
  "session_id": "string",
  "session_access_token": "string",
  "review_id": "string|null",
  "review_state": "READY|UPLOADING|ANALYZING_CONTRACT_TYPE|TYPE_SELECTION_REQUIRED|OUT_OF_SCOPE_CONFIRMATION_REQUIRED|READY_TO_REVIEW|QUEUED|REVIEWING|COMPLETED|FAILED|EXPIRED",
  "suggested_contract_type": "string|null",
  "selected_contract_type": "string|null",
  "selection_source": "suggested|candidate|manual|null",
  "retryable": true,
  "error_code": "string|null"
}
```

`session_id`는 업로드 요청 시 서버가 생성하고, `review_id`는 검토 요청 또는 재시도 요청 시 서버가 새로 생성한다. 로그인 없이 `session_access_token`을 가진 요청만 해당 세션과 검토 결과에 접근할 수 있다.

### 결과 항목

```json
{
  "status": "NONE|EXTRA|NO_MATCH|MISSING",
  "display_name": "string",
  "category": "string|null",
  "user_clause_id": "string|null",
  "standard_clause_id": "string|null",
  "user_clause": "string|null",
  "standard_clause": "string|null",
  "explanation": "string",
  "sources": []
}
```
