# 0724 최소 운영 기반: 로그, readiness, 운영 설정

- 상태: 승인됨
- 날짜: 2026-07-24
- 관련 ADR:
  - [API 설정과 의존성 구성](./0723-api-setting.md)
  - [파일형 SQLite와 영속성 계층](./0724-sqlite-persistence.md)

## 맥락

WorkShield는 비전공자를 포함한 5인 부트캠프 팀이 개발하는 단기 MVP다.
API 서버는 한 대만 운영하며 높은 트래픽, 다중 인스턴스, 장기 운영을
전제로 하지 않는다.

초기 검토에서는 구조화 로그, Prometheus 메트릭, APM, 요청 빈도 제한,
대시보드와 알림 등 일반적인 운영 기능을 폭넓게 고려했다. 그러나 현재
규모에서 이를 모두 도입하면 학습과 유지보수 비용이 핵심 사용자 흐름
구현보다 커질 수 있다.

FastAPI를 Uvicorn으로 실행하면 요청 경로와 상태 코드가 포함된 기본 access
로그와 error 로그가 제공된다. 다만 요청 ID, 검토 상태, 처리 시간 같은
애플리케이션 정보와 민감정보 비기록 정책은 별도로 구현해야 한다.

따라서 이번 결정은 다음 세 가지 공통 기능만 최소한으로 제공하는 것을
목표로 한다.

1. 안전한 애플리케이션 로그
2. SQLite와 MCP 준비 상태 확인
3. 운영 환경의 debug·OpenAPI·DB 쿼리 로그 통제

## 결정

### 1. 로깅

- Uvicorn의 기본 access/error 로그와 Python 표준 `logging`을 사용한다.
- Prometheus, APM, `structlog` 같은 추가 운영 패키지는 도입하지 않는다.
- 애플리케이션 이벤트는 한 줄 `key=value` 형식으로 기록한다.
- 공통 로그 함수가 받을 수 있는 필드는 다음으로 제한한다.
  - `event`
  - `request_id`
  - 세션 ID의 SHA-256 축약 해시
  - `review_id`
  - 처리 상태
  - 처리 시간
- 모든 HTTP 요청에 서버가 생성한 `request_id`, 응답 상태, 처리 시간을
  기록한다. 요청 경로와 요청 본문은 애플리케이션 이벤트 로그에 다시
  기록하지 않는다.
- 세션 ID 원문은 기록하지 않고 공통 로그 함수 안에서 해시한다.
- 예상하지 못한 예외는 안전한 500 응답으로 종료한다. 애플리케이션 로그에
  예외 메시지와 스택 트레이스를 남기지 않는다.
- 계약서, 조항, 프롬프트, 검토 결과, 대화 본문, API 키와 설정 객체 전체를
  로그에 기록하지 않는다.

향후 도메인 기능은 임의의 로그 포맷을 추가하지 않고 공통 `log_event()`를
사용한다.

### 2. 상태 확인

상태 확인 API의 책임을 다음과 같이 구분한다.

- `GET /health/live`
  - API 프로세스가 HTTP 요청에 응답할 수 있는지만 확인한다.
- `GET /health/ready`
  - SQLite에서 `SELECT 1`을 실행해 실제 연결 가능 여부를 확인한다.
  - FastAPI lifespan에서 WorkShield MCP 런타임이 초기화되었는지 확인한다.
  - 준비되지 않은 의존성이 있으면 HTTP 503을 반환한다.

readiness 요청마다 MCP 서버에 별도 도구 호출을 보내지 않는다. 서버 시작
시 수행한 capabilities handshake와 현재 런타임 존재 여부를 준비 상태의
기준으로 사용한다.

### 3. 운영 설정

- FastAPI debug는 `APP_DEBUG` 환경변수로 제어한다.
- OpenAPI 스키마와 Swagger/ReDoc 노출은 `API_DOCS_ENABLED`로 제어한다.
- 운영 기본값은 다음과 같다.
  - `APP_DEBUG=false`
  - `API_DOCS_ENABLED=false`
  - `DATABASE_ECHO=false`
- 운영 환경에서 debug 또는 SQLAlchemy 쿼리 로그를 활성화하면 설정 검증
  단계에서 애플리케이션 시작을 차단한다.
- `DEBUG`처럼 다른 도구와 충돌하기 쉬운 공용 환경변수 이름은 사용하지
  않는다.
- API 키는 기존 `SecretStr` 설정을 유지하고 설정 객체를 로그로 출력하지
  않는다.

## 제외한 범위

현재 MVP에서는 다음 항목을 구현하지 않는다.

- Prometheus와 `/metrics`
- APM과 분산 추적
- 별도 JSON 구조화 로그 패키지
- 운영 대시보드와 자동 알림
- IP·세션별 시간 단위 rate limiting

파일 크기 제한과 세션당 실행 중인 검토 수 제한은 운영 모니터링 기능이
아니라 각 사용자 흐름의 입력·상태 규칙으로 다룬다. 파일 업로드와 비동기
검토 실행기 담당 작업에서 각각 구현한다.

## 결과

장점:

- 별도 운영 인프라 없이 요청과 오류를 `request_id`로 추적할 수 있다.
- 민감한 계약 데이터가 애플리케이션 로그에 유입될 경로를 줄인다.
- 배포 환경에서 DB와 MCP 준비 상태를 간단히 확인할 수 있다.
- 팀원이 이해하고 테스트할 코드와 설정의 양을 작게 유지한다.

제약:

- 성공률, 처리 시간 분포, 현재 실행 수를 자동 집계하는 메트릭이 없다.
- 여러 서버의 로그를 중앙에서 조회하거나 요청 흐름을 분산 추적할 수 없다.
- 장애 알림과 장기 추세 분석은 수동 로그 확인에 의존한다.

## 검증

- 로그에 요청 경로의 민감한 query와 예외 메시지가 포함되지 않는지
  테스트한다.
- 세션 ID 원문 대신 해시가 기록되는지 테스트한다.
- SQLite와 MCP가 준비된 경우 readiness가 200을 반환하는지 테스트한다.
- SQLite 또는 MCP가 준비되지 않은 경우 readiness가 503을 반환하는지
  테스트한다.
- 운영 설정에서 debug와 DB 쿼리 로그 활성화가 거부되는지 테스트한다.
- OpenAPI 비노출 설정에서 `/openapi.json`, `/docs`, `/redoc`이 생성되지
  않는지 테스트한다.

## 재검토 조건

다음 중 하나가 발생하면 Prometheus, APM, 중앙 로그 수집 또는 rate
limiting 도입을 다시 검토한다.

- API 서버를 여러 대 실행한다.
- 공개 서비스의 운영 기간이나 사용자 수가 증가한다.
- 처리 시간과 성공률에 대한 대시보드 또는 자동 알림이 필요하다.
- 로그만으로 장애 원인과 요청 흐름을 추적하기 어렵다.
- 외부 APM을 도입하며 요청 본문·민감정보 수집 정책을 별도로 검증해야 한다.
