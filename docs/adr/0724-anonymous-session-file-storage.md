# 0724 익명 세션 접근 제어와 임시 파일 저장소

- 상태: 승인됨
- 날짜: 2026-07-24
- 관련 ADR:

  - [파일형 SQLite와 영속성 계층](./0724-sqlite-persistence.md)

## 맥락

WorkShield MVP는 로그인 기능 없이 계약서 업로드, 검토 진행 상태와 결과
조회를 제공한다. 따라서 `session_id`나 `review_id`만으로 리소스에 접근할
수 있으면 다른 사용자가 식별자를 획득하거나 추측해 계약서 처리 상태와
결과를 조회할 수 있다.

브라우저가 새로고침 후 화면을 복구하려면 세션과 검토 식별자는 보관해야
하지만, 리소스 접근 권한을 나타내는 비밀값을 JavaScript 저장소에 두면
XSS나 디버그 출력으로 노출될 가능성이 커진다.

업로드 파일은 검토와 재시도 동안 일시적으로 필요하다. 기존
`storage_path`를 Application Service나 MCP 연동 코드가 직접 다루면 사용자
파일명이 저장 경로에 포함되거나 경로 순회, 중복 삭제, 비정상 종료 후
잔존 파일 같은 문제가 여러 계층에 분산된다.

운영 정책은 다음을 요구한다.

- 사용자별 세션·파일·결과 격리
- 마지막 사용자 활동 후 30분 TTL
- 재시도 불가능 실패와 만료 세션의 파일 삭제
- 재시도 가능 실패의 TTL 내 원본 파일 보존
- 서버 재시작 후 잔존 파일 정리
- 존재하지 않는 리소스와 접근 불가 리소스의 동일한 404 처리

## 결정

### 1. 익명 세션 접근 토큰

- 세션 생성 시 `secrets.token_urlsafe(32)`로 최소 256비트 엔트로피의 접근
  토큰을 생성한다.
- 원본 토큰은 생성 응답의 본문에 포함하지 않고
  `workshield_session` HttpOnly Cookie로만 전달한다.
- 운영 환경의 Cookie에는 `Secure`를 적용한다.
- Cookie 범위는 `/api/v1`, SameSite 정책은 `Lax`로 한다.
- 프론트엔드는 토큰을 `localStorage`나 `sessionStorage`에 저장하지 않는다.
- DB에는 원본 토큰이 아닌 SHA-256 해시만 저장한다.
- 접근 토큰 원본과 Cookie 값은 애플리케이션 로그에 기록하지 않는다.

접근 토큰은 충분한 엔트로피를 가진 서버 생성 난수이므로 비밀번호용 느린
해시 대신 SHA-256을 사용한다. 토큰 해시는 비밀번호와 달리 사용자 선택
값이 아니며 현실적인 사전 대입 공격 대상이 아니다.

### 2. 리소스 소유권 검증

- 세션 조회는 `session_id + access_token_hash` 조합으로 수행한다.
- 검토 조회는 `review_id`와 상위 세션의 `access_token_hash`를 조인해 한
  번에 수행한다.
- Router와 Application Service는 `session_id` 또는 `review_id`만으로
  사용자 리소스를 조회하지 않는다.
- 세션 API는 `OwnedReviewSessionDep`, 검토 API는 `OwnedReviewDep`를
  공통으로 사용한다.
- BE-B의 상태, 결과, SSE, 재시도, grounding, chat, suggestions API도
  동일한 검토 접근 Dependency를 사용한다.

오류 판정 순서는 다음과 같다.

1. Cookie가 없거나 ID와 토큰 해시 조합이 일치하지 않으면
   `404 RESOURCE_NOT_FOUND`
2. 소유권이 확인된 뒤 세션 또는 검토가 만료되었으면
   `410 SESSION_EXPIRED`

존재 여부를 먼저 조회한 뒤 권한을 확인하지 않는다. 이를 통해 다른
세션에서 리소스의 존재 여부나 만료 여부를 구분할 수 없게 한다.

### 3. FileStorage 경계

- 사용자 파일은 `FileStorage` Protocol을 통해서만 저장·열기·삭제한다.
- MVP 구현체는 로컬 파일 시스템을 사용하는 `LocalFileStorage`다.
- 저장소가 서버 생성 난수와 검증된 확장자로 `storage_key`를 만든다.
- 사용자 파일명은 저장 경로 구성에 사용하지 않는다.
- DB와 Domain Entity에는 실제 경로가 아닌 `storage_key`만 저장한다.
- 파일 저장은 임시 파일 기록 후 atomic replace로 완료한다.
- 삭제는 파일이 이미 없어도 성공하는 멱등 연산으로 구현한다.
- storage key 검증으로 절대 경로와 `..` 경로 순회를 차단한다.
- stdio MCP가 로컬 경로를 요구할 때도 `FileStorage.local_path()`가
  제공하는 컨텍스트 안에서만 경로를 사용한다.

Application Service와 BE-B는 저장 루트를 조합하거나 `open()`,
`Path.unlink()`를 직접 호출하지 않는다.

### 4. 파일 보존과 정리

- 재시도 가능 실패에서는 세션 TTL까지 파일을 보존한다.
- 재시도 불가능 실패에서는 파일을 즉시 삭제하고 `storage_key`를 비운다.
- 만료 세션은 파일을 삭제하고 세션·검토 상태를 `EXPIRED`로 변경한다.
- 만료 시 검토의 `progress`, `result`, `error`를 제거한다.
- 소유자의 만료 접근에 410을 반환할 수 있도록 최소 만료 표식은 DB에
  유지한다.
- `QUEUED` 또는 `REVIEWING` 검토가 있는 세션은 만료 정리하지 않는다.
- 서버 시작 시 DB가 참조하지 않는 고아 파일과 중단된 업로드 조각을
  정리한다.
- 서버 실행 중에는 기본 60초 간격으로 만료 세션을 정리한다.
- 실행 중 정리에서는 저장 직후 DB 반영 전 파일과 경쟁하지 않도록 고아
  파일 정리를 수행하지 않는다.

파일 시스템 삭제와 SQLite 갱신은 하나의 원자적 트랜잭션이 될 수 없다.
따라서 파일 삭제와 정리 작업 전체를 반복 호출할 수 있도록 설계한다.

### 5. 설정과 스키마

- 세션 TTL 기본값은 운영 정책에 맞춰 1,800초로 한다.
- 정리 주기 기본값은 60초로 한다.
- `review_sessions.storage_path` 대신 nullable `storage_key`를 사용한다.
- 초기 개발 DB에 `storage_key`가 없으면 서버 시작 시 컬럼을 추가한다.
- 기존 `storage_path` 값은 새 저장소 key로 신뢰하거나 자동 변환하지
  않는다.

## 고려한 대안

### `session_id` 자체를 비밀값으로 사용

응답, URL, 브라우저 상태 복구와 로그 추적에 사용하는 공개 식별자와 접근
권한을 분리할 수 없어 채택하지 않았다.

### 토큰을 응답 본문에 포함하고 Web Storage에 저장

구현은 단순하지만 JavaScript와 XSS가 토큰에 접근할 수 있으므로 채택하지
않았다.

### 토큰 원본을 DB에 저장

DB 노출 시 활성 세션에 즉시 접근할 수 있으므로 채택하지 않았다.

### 접근 불가 리소스에 401 또는 403 반환

리소스 존재 여부를 구분할 수 있으므로 사용자 리소스 조회에는 채택하지
않았다. 로그인·관리자 권한이 도입되면 인증 자체의 401과 소유권 은닉
404를 별도로 재검토한다.

### Application Service에서 실제 파일 경로 직접 사용

저장 기술과 보안 검증, 삭제 정책이 여러 도메인에 중복되므로 채택하지
않았다.

## 결과

장점:

- `session_id`와 `review_id`만으로 사용자 리소스에 접근할 수 없다.
- DB가 노출되어도 저장된 해시만으로 활성 세션 Cookie를 재현할 수 없다.
- 세션과 검토 API가 동일한 소유권 검증을 재사용한다.
- BE-B가 파일 시스템 경로와 삭제 방식에 의존하지 않는다.
- 파일 삭제와 정리 작업을 안전하게 재시도할 수 있다.
- 만료된 소유자와 접근 불가 사용자의 오류를 정책대로 구분한다.

제약:

- 브라우저 Cookie 하나가 현재 익명 세션 접근 권한을 나타내므로 같은
  브라우저에서 완전히 독립된 여러 익명 세션을 동시에 사용하는 기능은
  제공하지 않는다.
- HttpOnly Cookie를 사용하므로 프론트와 API의 CORS와
  credentials 설정이 정확해야 한다.
- 로컬 파일 시스템은 여러 API 서버가 동시에 공유하는 배포를 지원하지
  않는다.
- SQLite와 파일 시스템 사이에 분산 트랜잭션이 없으므로 정리 작업의
  멱등성에 의존한다.
- 현재 TTL은 세션 만료 시각을 기준으로 동작하며, 모든 사용자 활동에서
  TTL을 연장하는 정책은 각 Application Service가 명시적으로 적용해야
  한다.

## 검증

- 서로 다른 토큰이 서로 다른 해시를 생성하는지 테스트한다.
- DB에 접근 토큰 원본이 아닌 해시만 저장되는지 테스트한다.
- Cookie가 HttpOnly이며 응답 본문에 토큰이 없는지 테스트한다.
- Cookie 없이 `session_id` 또는 `review_id`만 요청하면 404인지 테스트한다.
- 다른 브라우저 토큰으로 세션과 검토를 요청하면 동일한 404인지
  테스트한다.
- 소유한 만료 세션은 410, 타인의 만료 세션은 404인지 테스트한다.
- 서버 생성 storage key와 경로 순회 차단을 테스트한다.
- 파일 삭제를 여러 번 호출해도 오류가 없는지 테스트한다.
- 재시도 가능·불가능 실패의 파일 보존 차이를 테스트한다.
- 진행 중 리뷰가 만료 정리에서 제외되는지 테스트한다.
- 만료 데이터와 고아·불완전 파일 정리를 테스트한다.

## 재검토 조건

다음 중 하나가 발생하면 세션·Cookie·저장소 결정을 재검토한다.

- 로그인 또는 조직별 계정 체계를 도입한다.
- 하나의 브라우저에서 여러 익명 세션을 동시에 유지해야 한다.
- 프론트와 API가 서로 다른 사이트에 배포되어 `SameSite=None`이 필요하다.
- API 서버를 여러 대 실행하거나 컨테이너의 로컬 디스크를 공유할 수 없다.
- S3 등 외부 Object Storage를 도입한다.
- 만료 tombstone의 장기 보관량이 운영상 문제가 된다.
- 모든 사용자 활동을 기준으로 sliding TTL을 일관되게 적용해야 한다.
