* 상태: Proposed

## 1. 배경

WorkShield는 IT·SW 계약서를 표준계약서와 조항별로 비교하고, 표준 대비 차이와 누락 가능성, 주의 문구 유사 신호, 관련 법령 원문을 제공하는 서비스다.

MCP는 검색·재정렬·분류를 통해 다음과 같은 구조화된 검토 후보를 생성한다.

* `NONE`
* `EXTRA`
* `NO_MATCH`
* `MISSING`
* 주의 문구 유사 신호
* 대응 표준조항
* 관련 법령 원문

이 결과는 법률 자문이나 위법·합법 판단이 아니며, 사용자가 우선 확인할 부분을 좁혀주는 참고 자료다.

LLM은 MCP 결과를 기반으로 다음 기능을 제공한다.

* 검토 결과에 대한 자연어 설명
* 사용자 질문에 대한 근거 기반 답변
* 계약 조항 수정·협의 문구 초안 생성

현재 로컬 LLM 운영을 위해 Ollama를 사용할 예정이지만, 모델 실행 환경, 구조화 출력 안정성, 출처 검증, 장애 처리 및 보안 통제가 충분히 검증되지 않았다.

## 2. 프로젝트 제약

다음 조건을 고려하여 기술 결정을 내린다.

* 5인 부트캠프 팀 프로젝트
* 상업 목적이 아닌 단기 MVP
* 운영 기간은 최대 약 3일
* 개발자는 저사양 개인 노트북을 사용
* 웹·API·MCP는 Cloudflare 기반으로 배포
* 생성형 LLM은 RunPod GPU 환경에서 실행
* 제한된 운영 예산
* 계약서와 대화 내용은 민감정보로 취급
* 외부 LLM으로 자동 전환하지 않음
* 법률적 결론을 생성하거나 표시하지 않음

## 3. 검토한 주요 위험

### 3.1 실행 환경 위험

모델명만 정하고 다음 항목을 고정하지 않으면 실제 운영 가능성을 판단할 수 없다.

* 정확한 모델 태그
* 양자화 방식
* GPU 및 VRAM
* 컨텍스트 길이
* 최대 출력 토큰
* 동시 요청 수
* 모델 로딩 및 재시작 방식

긴 계약 조항과 법령 원문을 입력할 경우 추론 지연 또는 GPU OOM이 발생할 수 있다.

### 3.2 개발·운영 환경 불일치

OpenAI 모델로 기능을 개발한 후 운영 시점에 Ollama 모델로 전환하면 다음 차이가 발생할 수 있다.

* JSON 형식 준수율
* 한국어 답변 품질
* 시스템 지시 준수
* 긴 입력 처리 능력
* 응답 중단 및 반복 출력
* 프롬프트 인젝션 방어 성능

따라서 운영에 사용하는 Ollama 모델을 기준으로 통합 검증해야 한다.

### 3.3 구조화 출력 및 허위 출처 위험

LLM 자유 응답을 그대로 사용자에게 노출하면 다음 문제가 발생할 수 있다.

* 존재하지 않는 조항 식별자 생성
* 다른 세션의 조항 참조
* 존재하지 않는 법령 조문 생성
* MCP 상태값 임의 변경
* 필수 필드 누락
* 법률적 단정 표현 생성
* 계약서에 없는 금액·기한·비율 생성

모든 LLM 출력은 백엔드 검증을 통과한 경우에만 표시해야 한다.

### 3.4 프롬프트 인젝션 위험

계약서 본문과 법령 원문은 신뢰할 수 없는 입력이다.

문서에 포함된 지시문이 시스템 명령으로 해석되지 않도록 계약서와 법령을 읽기 전용 데이터로 격리해야 한다.

### 3.5 민감정보 및 로그 위험

계약서 원문, 사용자 조항, 법령 컨텍스트, 대화 내용과 프롬프트가 다음 위치에 기록될 수 있다.

* FastAPI 로그
* Ollama 로그
* 컨테이너 표준 출력
* Cloudflare 로그
* APM 및 트레이싱 도구
* 예외 스택의 지역 변수
* 임시 파일 저장소

운영 전 실제 로그와 삭제 동작을 검증해야 한다.

## 4. 결정

### 4.1 전체 결정

다음 구조를 WorkShield의 로컬 LLM 기본 아키텍처로 채택한다.

```text
사용자 요청
→ 백엔드 요청 범위 및 세션 검증
→ 백엔드가 MCP 도구 호출
→ MCP 결과와 허용 근거 구성
→ LLM용 제한 컨텍스트 생성
→ Ollama 모델 호출
→ 구조화 출력 검증
→ 출처 및 정책 검증
→ 검증된 응답만 사용자에게 반환
```

MCP는 판정과 근거 생성을 담당하고, LLM은 설명과 문구 초안만 담당한다.

### 4.2 운영 모델과 실행 환경

운영 후보는 다음과 같이 고정한다.

* 실행 엔진: Ollama
* 운영 후보 모델: Qwen3.5 9B 계열
* 양자화: Q4 계열
* 운영 GPU: VRAM 24GB급 RunPod GPU
* 초기 동시 처리 수: 1
* 기본 생성 모드: 비추론 모드
* 최대 출력: 약 1,000 토큰 이내
* 외부 LLM 자동 폴백: 사용하지 않음

정확한 Ollama 태그, 양자화 파일 및 컨텍스트 크기는 운영 검증 이슈에서 실제 실행 결과를 확인한 후 고정한다.

개발 중 더 작은 모델을 사용할 수 있지만 다음 테스트는 반드시 운영 후보 모델로 수행한다.

* 출력 스키마 검증
* 한국어 품질 평가
* 프롬프트 인젝션 테스트
* 성능 및 메모리 테스트
* 장애 및 재시도 테스트

### 4.3 MCP와 LLM 역할 분리

MCP는 다음 데이터를 결정하는 유일한 주체다.

* 계약 유형 범위 판별
* 계약서 조항 파싱
* 표준조항 검색 및 선택
* `NONE`, `EXTRA`, `NO_MATCH`, `MISSING` 상태
* 매칭 점수
* 주의 문구 패턴
* 표준조항 식별자
* 법령 원문과 출처

LLM은 다음 값을 생성하거나 변경할 수 없다.

* MCP 상태 코드
* 조항 카테고리
* 매칭 점수
* 표준조항 식별자
* 법령 출처 식별자
* 주의 문구 패턴 코드
* 계약 유형

LLM이 수행할 수 있는 작업은 다음으로 제한한다.

* MCP 결과에 대한 중립적 설명
* 제공된 근거 내 질문 답변
* 충분한 근거가 있는 조항의 수정·협의 문구 초안 작성
* 답변 한계와 추가 확인 필요사항 표시

### 4.4 MCP 호출 주체

LLM은 MCP 도구를 직접 호출하지 않는다.

백엔드가 사용자 요청과 현재 상태에 따라 필요한 도구를 선택한다.

주요 도구는 다음과 같다.

* `assess_contract_scope`
* `parse_contract_clauses`
* `review_contract_candidates`
* `classify_clause_candidate`
* `get_category_grounding`
* `list_contract_types`
* `list_categories`
* `list_toxic_pattern_details`

LLM에는 백엔드가 이미 조회하고 검증한 읽기 전용 데이터만 전달한다.

LLM에 다음 권한을 부여하지 않는다.

* 파일 시스템 접근
* 전체 MCP 도구 탐색
* 임의 MCP 호출
* 임의 법령 검색
* 외부 URL 조회
* 다른 사용자 세션 접근
* 계약서 전체 재검토 실행

### 4.5 LLM 입력 컨텍스트

전체 계약서 원문을 기본 입력으로 사용하지 않는다.

백엔드는 요청마다 필요한 정보만 포함하는 컨텍스트를 구성한다.

```text
1. 시스템 정책
2. 사용자의 현재 질문
3. 관련 사용자 조항
4. MCP 검토 결과
5. 선택된 표준조항
6. 필요한 법령 원문
7. 제한된 최근 대화
```

컨텍스트 생성 결과에는 다음 메타데이터를 유지한다.

* 예상 입력 토큰 수
* 포함한 근거 식별자
* 제외한 근거 식별자
* 잘림 여부
* 적용한 컨텍스트 제한값

모델의 최대 컨텍스트와 별도로 실제 요청 입력은 가능한 한 4K~8K 토큰 범위를 목표로 한다.

### 4.6 출처 식별 방식

LLM에 실제 내부 식별자를 직접 생성하도록 요구하지 않는다.

백엔드는 현재 요청에서 사용할 근거에 임시 출처 키를 부여한다.

```json
{
  "SRC_1": {
    "type": "USER_CLAUSE",
    "id": "현재 세션 사용자 조항 ID"
  },
  "SRC_2": {
    "type": "STANDARD_CLAUSE",
    "id": "표준조항 ID"
  },
  "SRC_3": {
    "type": "LAW",
    "id": "법령 출처 ID"
  }
}
```

LLM은 `SRC_1`, `SRC_2`와 같은 키만 반환한다.

백엔드는 다음을 검증한다.

* 현재 요청에 등록된 출처 키인지
* 현재 세션의 데이터인지
* 실제 MCP 결과에 존재하는지
* 출처 유형이 답변 내용과 일치하는지

검증되지 않은 출처가 포함되면 응답 전체를 표시하지 않는다.

### 4.7 구조화 출력

챗봇 답변과 문구 초안은 서로 다른 출력 스키마를 사용한다.

#### 챗봇 답변

```json
{
  "answer": "근거 기반 설명",
  "refused": false,
  "source_keys": ["SRC_1", "SRC_2"],
  "limitations": [],
  "tool_status": "OK"
}
```

#### 수정·협의 문구 초안

```json
{
  "suggestion": "제안 문구",
  "purpose": "변경 목적",
  "major_changes": ["주요 변경사항"],
  "source_keys": ["SRC_1", "SRC_2"],
  "placeholders": [],
  "limitations": []
}
```

출력 처리 순서는 다음과 같다.

```text
LLM 원본 응답
→ JSON 파싱
→ Pydantic 스키마 검증
→ 출처 allowlist 검증
→ MCP 값 변경 여부 검증
→ 금지 표현 및 값 생성 검증
→ 사용자 응답 반환
```

검증 전 원본 출력은 사용자에게 노출하지 않는다.

### 4.8 문구 초안 생성 조건

문구 초안은 충분한 근거가 있는 경우에만 생성한다.

기본 허용 조건은 다음과 같다.

* 사용자 조항이 존재함
* `match.status == CANDIDATE_SELECTED`
* 대응 표준조항이 존재함
* 현재 세션의 근거가 확보됨

일반적으로 `NONE` 또는 `EXTRA` 상태에서 대응 표준조항이 선택된 경우를 대상으로 한다.

다음 상황에서는 초안 생성을 차단하거나 근거 부족으로 처리한다.

* `NO_MATCH`
* `MISSING`
* 대응 표준조항 없음
* 법령 또는 조항 출처 검증 실패
* 입력 길이 초과로 핵심 근거가 제외됨

MISSING은 사용자 원문 조항이 존재하지 않으므로 별도의 누락 조항 추가 기능으로 취급한다.

### 4.9 수치와 계약 조건 생성 제한

LLM은 계약서와 근거에 없는 다음 값을 새로 생성할 수 없다.

* 금액
* 비율
* 날짜
* 기간
* 지체상금
* 손해배상 한도
* 지급 조건
* 영업상 조건

필요한 값이 근거에 없으면 다음과 같은 자리표시자를 사용한다.

```text
[금액 확인 필요]
[기간 협의 필요]
[비율 입력 필요]
```

### 4.10 오류와 재시도

자동 재시도는 최대 1회로 제한한다.

| 오류           | 처리               |
| ------------ | ---------------- |
| 일시적 연결 실패    | 1회 재시도 가능        |
| 잘못된 JSON     | 보정 프롬프트로 1회 재시도  |
| 출처 검증 실패     | 1회 재생성 후 실패      |
| GPU OOM      | 동일 입력 자동 재시도 금지  |
| 입력 길이 초과     | 컨텍스트 축소 후에만 재실행  |
| 모델 미설치       | 즉시 실패            |
| Ollama 연결 불가 | 즉시 오류 상태 반환      |
| 빈 응답         | 1회 재시도 후 실패      |
| 타임아웃         | 작업 결과 폐기 후 오류 처리 |

재시도 요청에도 동일한 세션과 요청 식별자를 사용하며, 중복 GPU 작업을 방지하기 위해 idempotency를 적용한다.

### 4.11 상태 처리

LLM 기능은 다음 상태를 사용한다.

#### 챗봇

* `DISABLED`
* `READY`
* `ANSWERING`
* `ANSWERED`
* `REFUSED`
* `ERROR`
* `EXPIRED`

#### 문구 초안

* `IDLE`
* `SOURCE_SELECTED`
* `GENERATING`
* `GENERATED`
* `INSUFFICIENT_GROUNDING`
* `ERROR`

정책 범위 밖 질문이나 법률적 결론 요청은 `REFUSED`로 처리한다.

근거가 부족한 경우 임의 답변 대신 `REFUSED` 또는 `INSUFFICIENT_GROUNDING`을 반환한다.

### 4.12 보안 및 데이터 처리

다음 내용을 로그에 기록하지 않는다.

* 계약서 원문
* 사용자 조항 원문
* 법령 원문
* 전체 프롬프트
* LLM 원본 응답
* 사용자 대화 본문

로그에는 다음 운영 정보만 기록한다.

* 요청 식별자
* 세션 식별자의 비가역 값
* 모델 버전
* 처리 시간
* 입력·출력 토큰 추정치
* 성공·실패 상태
* 오류 유형
* 재시도 횟수

계약서와 생성 결과는 완료, 실패, 취소 또는 TTL 만료 시 삭제한다.

외부 LLM으로 자동 전환하지 않는다.

## 5. 검토한 대안

### 5.1 OpenAI를 운영 기본 모델로 사용

장점:

* 높은 구조화 출력 안정성
* 개발 편의성
* 운영 관리 부담 감소

채택하지 않은 이유:

* 계약서 데이터의 외부 전송 부담
* 자체 호스팅 운영 방침과 불일치
* 단기 프로젝트의 운영 정책과 맞지 않음

필요한 경우에도 사용자 동의와 별도 정책 없이 자동 전환하지 않는다.

### 5.2 로컬 4B 모델만 사용

장점:

* 낮은 GPU 요구사항
* 빠른 응답
* 개발 환경 구성 용이

채택하지 않은 이유:

* 한국어 계약 문장의 설명 품질과 구조화 출력 안정성이 충분하지 않을 가능성
* 운영 품질 검증 결과에 따라 보조 또는 개발용으로만 사용

### 5.3 20B 이상 대형 모델 사용

장점:

* 답변 품질 향상 가능성

채택하지 않은 이유:

* 제한된 예산과 운영 기간
* GPU 비용과 메모리 요구량 증가
* 부트캠프 MVP 범위를 초과하는 운영 복잡성

### 5.4 LLM이 직접 MCP 도구를 선택

장점:

* 유연한 에이전트형 질의 처리

채택하지 않은 이유:

* 불필요하거나 잘못된 도구 호출 가능성
* 세션 외 데이터 접근 위험
* 프롬프트 인젝션 공격 면 증가
* 호출 경로와 비용을 예측하기 어려움
* MCP 판정과 LLM 설명의 책임 경계가 흐려짐

## 6. 결정에 따른 결과

### 긍정적 결과

* MCP의 결정론적 판정과 LLM 설명을 분리할 수 있다.
* 허위 출처와 세션 간 데이터 혼입을 백엔드에서 차단할 수 있다.
* 생성형 모델 장애가 계약 검토 결과 자체를 변경하지 않는다.
* 모델 교체 시에도 애플리케이션 입출력 계약을 유지할 수 있다.
* 단기 MVP에 필요한 운영 범위를 제한할 수 있다.

### 부정적 결과

* 백엔드에 컨텍스트 빌더와 출력 검증 로직이 추가된다.
* LLM 자유도가 줄어든다.
* 일부 질문은 근거 부족으로 거절될 수 있다.
* Ollama 및 GPU 장애를 별도로 운영해야 한다.
* 취소된 요청의 GPU 연산이 즉시 중단되지 않을 수 있다.

## 7. 잔여 위험

다음 위험은 본 결정만으로 제거되지 않는다.

* 운영 모델의 실제 한국어 품질
* 긴 입력에서의 처리 지연
* GPU OOM
* 구조화 출력 실패
* 프롬프트 인젝션 우회
* Ollama 또는 컨테이너 로그의 민감정보 기록
* 사용자 취소 후 GPU 연산 지속
* 모델 업데이트에 따른 출력 품질 변화

잔여 위험은 별도의 운영·검증 GitHub Issue에서 테스트한다.

## 8. 승인 조건

다음 조건을 모두 충족하면 ADR 상태를 `Accepted`로 변경한다.

* 정확한 모델 태그와 양자화 고정
* 운영 GPU에서 모델 정상 로딩
* 최대 입력 조건에서 OOM 미발생
* 구조화 출력 최종 성공률 99% 이상
* 검증되지 않은 응답 노출 0건
* 존재하지 않는 출처 허용 0건
* 다른 세션의 출처 허용 0건
* MCP 상태 변경 허용 0건
* 근거에 없는 금액·날짜·비율 생성 0건
* 프롬프트 인젝션 정책 우회 0건
* 법률적 단정 표현 노출 0건
* 계약서·프롬프트 본문 로그 기록 0건
* 필수 장애 시나리오 검증 완료

## 9. 재검토 조건

다음 상황이 발생하면 이 결정을 다시 검토한다.

* 운영 GPU 또는 예산 변경
* 모델 또는 양자화 변경
* 동시 사용자 수 증가
* 운영 기간 연장
* 외부 공개 서비스로 전환
* 법률 자문 수준의 기능 요구
* LLM이 직접 도구를 호출해야 하는 요구 발생
* 현재 모델이 품질 또는 성능 합격 기준을 충족하지 못함

## 10. 최종 결정 요약

WorkShield는 MCP를 검토 판정과 근거의 유일한 원천으로 사용한다.

로컬 LLM은 백엔드가 선별한 현재 세션의 근거만 입력받아 설명과 문구 초안을 생성한다. 모든 출력은 구조화 스키마, 출처 allowlist, 상태 불변성, 금지 표현 및 수치 생성 검증을 통과한 경우에만 사용자에게 표시한다.

Ollama 기반 Qwen3.5 9B Q4 계열과 24GB급 GPU를 운영 후보로 사용하되, 운영·검증 체크리스트를 모두 통과하기 전까지 본 결정의 상태는 `Proposed`로 유지한다.
